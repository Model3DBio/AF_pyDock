# Generating 3D Models with AlphaFold

This directory contains ready-to-use inputs and helpers for generating and
preparing protein-protein complex models:

* `run_alphafold2_multimer.sh`: creates an input FASTA file and runs the selected
  ColabFold / AlphaFold2-Multimer versions, using v2 and v3 by default and saving
  intermediate recycles. The final ranked models are relaxed with Amber on the
  GPU during the run.
* `relax_alphafold2_models.sh`: relaxes saved unrelaxed AlphaFold2 recycle PDB
  files with `colabfold_relax`, using Greasy for parallel execution.
* `json_files_for_alphafold3_server/`: prepared protein-only JSON jobs for
  manual submission through AlphaFold Server.
* `run_alphafold3.sh`: creates a protein-only AlphaFold 3 JSON input and runs
  a local AlphaFold 3 installation.
* `cif_to_pdb.py`: converts AF3 CIF models to PDB files.

The scripts are configured with environment variables and include a default
`4POU` example where applicable.

The scripts can be launched from the repository root, from this section
directory or from another working directory. Case-study inputs and results are
always read from or written to `4_Case_Studies/<CASE>/`; AF2 and AF3 results are
stored in the `AF2/` and `AF3/` subdirectories, respectively.

The JSON files are inputs, whereas the AlphaFold structures and confidence
files are generated results. No generated AF3 results are stored inside
`json_files_for_alphafold3_server/`.

## Conda environments and executable checks

Activate the appropriate software environment before starting a local
calculation. The examples use the default names `Alphafold2` and `Alphafold3`,
but another Conda environment or a non-Conda installation can be used when the
required commands are available through `PATH` or the variables documented
below. The scripts report a warning for a different active environment and
stop if a required executable cannot be found.

---

## Workflow Summary

```text
Sequences
  -> AF2-Multimer FASTA / local AF3 JSON / AlphaFold Server JSON
  -> AlphaFold model generation (local installation or external server)
  -> optional AF2 recycle relaxation / AF3 CIF-to-PDB conversion
  -> PDB model folders for downstream pyDock scoring
```

For the server route, this repository prepares the job definitions, but job
submission, AF3 inference and result download still take place on AlphaFold
Server. For local AF3, `run_alphafold3.sh` performs input generation and invokes
the configured local installation.

## Script 1: `run_alphafold2_multimer.sh`

This script runs ColabFold with one or more selectable AlphaFold2-Multimer model
versions:

* `alphafold2_multimer_v1` (legacy, opt-in)
* `alphafold2_multimer_v2` (selected by default)
* `alphafold2_multimer_v3` (selected by default)

> **Version-selection note:** AlphaFold2-Multimer v1 is retained to reproduce
> the chapter workflow and existing case-study outputs. For new predictions,
> v2 or v3 should normally be preferred. In the models tested for this
> repository, v1 structures showed severe steric clashes more frequently,
> which can make downstream relaxation substantially slower.

It uses the following ColabFold options:

| Option | Purpose |
| --- | --- |
| `--save-recycles` | Save intermediate recycle structures. |
| `--model-type alphafold2_multimer_v*` | Run each version selected through `AF2_MULTIMER_VERSIONS`. |
| `--recycle-early-stop-tolerance 0` | Avoid early stopping before the requested recycles. |
| `--rank multimer` | Rank models with the multimer score. |
| `--use-dropout` | Increase model diversity during inference. |
| `--num-recycle 20` | Run up to 20 recycling iterations. |
| `--amber` | Relax all final ranked models generated for each selected AF2 version. |
| `--use-gpu-relax` | Perform the final-model Amber relaxation on the GPU. |

The relaxed final ranked models are generated directly by this script, as in
the supplied case-study archives. The intermediate recycle structures saved by
`--save-recycles` remain unrelaxed; use `relax_alphafold2_models.sh` when the
complete relaxed recycle ensemble is required for downstream scoring.

### AF2 input variables

| Variable | Default | Purpose |
| --- | --- | --- |
| `CASE` | `4POU` | Case identifier used for folders and file names. |
| `SEQ` | 4POU VHH/RNase A sequence | Protein sequences. Separate chains with `:`. |
| `AF2_CONDA_ENV` | `Alphafold2` | Expected Conda environment name. A different environment or a non-Conda installation can still be used if the executable is available. |
| `COLABFOLD_BATCH` | `colabfold_batch` | ColabFold executable name or explicit executable path. |
| `AF2_MULTIMER_VERSIONS` | `v2 v3` | Space-separated selection of `v1`, `v2` and/or `v3`. Duplicate or unsupported values stop the script before inference. |

### AF2 usage example

From this section directory:

```bash
conda activate Alphafold2

export CASE="MyComplex"
export SEQ="AAAAABBBB:CCCCDDDD"

bash run_alphafold2_multimer.sh
```

The equivalent command from the repository root is:

```bash
CASE="MyComplex" SEQ="AAAAABBBB:CCCCDDDD" \
  bash 3.1_Generating_3D_Models_for_Protein_Protein_Complexes_with_AlphaFold/run_alphafold2_multimer.sh
```

To run only AlphaFold2-Multimer v3, set:

```bash
AF2_MULTIMER_VERSIONS="v3" CASE="MyComplex" SEQ="AAAAABBBB:CCCCDDDD" \
  bash 3.1_Generating_3D_Models_for_Protein_Protein_Complexes_with_AlphaFold/run_alphafold2_multimer.sh
```

### AF2 generated files

The script writes the following case-study files:

```text
4_Case_Studies/
└── ${CASE}/
    └── AF2/
        ├── ${CASE}.fasta
        ├── ${CASE}.colabfold.v2/
        │   ├── *_relaxed_rank_*.pdb
        │   ├── *_unrelaxed_rank_*.pdb
        │   └── *_unrelaxed_rank_*.r<N>.pdb
        └── ${CASE}.colabfold.v3/
            ├── *_relaxed_rank_*.pdb
            ├── *_unrelaxed_rank_*.pdb
            └── *_unrelaxed_rank_*.r<N>.pdb
```

The directories correspond to the selected versions. With the default
`AF2_MULTIMER_VERSIONS="v2 v3"`, no v1 directory is generated; selecting `v1`
adds `${CASE}.colabfold.v1/`.

The generated FASTA identifies the case and contains the chain sequences
provided through `SEQ`. Each selected version directory contains the relaxed
final ranked models, their unrelaxed counterparts and the unrelaxed recycle
snapshots (`.r<N>.pdb`).

## Script 2: `relax_alphafold2_models.sh`

`run_alphafold2_multimer.sh` relaxes the final ranked models, but the saved
recycle snapshots remain unrelaxed. Use this post-processing script when the
downstream pyDock workflow should score the complete relaxed recycle ensemble.

By default, if neither `CASE` nor `INPUT_DIR` is defined, it scans all case
study AF2 folders matching:

```text
<repo_root>/4_Case_Studies/*/AF2
```

If `CASE` is defined, it uses:

```text
<repo_root>/4_Case_Studies/${CASE}/AF2
```

If `INPUT_DIR` is defined, that directory is used directly.

### AF2 relaxation variables

| Variable | Default | Purpose |
| --- | --- | --- |
| `CASE` | empty | Optional case identifier, for example `4POU` or `2FJG`. |
| `INPUT_DIR` | empty | Custom AF2 directory to scan. Overrides `CASE`; relative values are resolved from the repository root. |
| `INPUT_ROOT` | `<repo_root>/4_Case_Studies` | Base directory used with `CASE` or all case studies; relative values are resolved from the repository root. |
| `PDB_PATTERN` | `*_unrelaxed_*.pdb` | PDB filename pattern to relax. |
| `OVERWRITE` | `0` | Skip existing relaxed outputs unless set to `1`. |
| `AF2_RELAX_VERSIONS` | `v2 v3` | Space-separated AF2-Multimer versions to relax. Standard `*.colabfold.v1/`, `.v2/` and `.v3/` directories are filtered accordingly. |
| `AF2_CONDA_ENV` | `Alphafold2` | Expected Conda environment name. |
| `COLABFOLD_RELAX` | `colabfold_relax` | Relaxation executable name or explicit executable path written to the Greasy task file. |
| `GREASY_HOME` | `/path/to/software/GREASY_2.2` | Greasy installation prefix. |
| `GREASY` | `${GREASY_HOME}/bin` | Directory containing the public `greasy` wrapper. An explicitly defined value takes precedence over `GREASY_HOME`. |
| `GREASY_NWORKERS` | `4` | Number of parallel Greasy workers. This value must be adjusted to the available GPU memory. |
| `RELAX_TIMEOUT` | `8m` | Maximum duration of each relaxation task, using GNU `timeout` duration syntax. |
| `RUN_GREASY` | `1` | Set to `0` to create the task file without launching Greasy. |
| `JOB_FILE` | `<section_dir>/${CASE:-case_studies}.af2_relax.greasy` | Greasy task file. Relative overrides are also resolved from the section directory. |

Relaxed output names are generated by replacing `_unrelaxed_` with
`_relaxed_` in each PDB filename, so each result remains beside its source model
inside `4_Case_Studies/<CASE>/AF2/`. The Greasy task file is kept in this
section directory rather than in the caller's working directory.

`AF2_RELAX_VERSIONS` controls relaxation independently of model generation.
With its default `v2 v3`, models under standard `*.colabfold.v1/` directories
are skipped even when legacy v1 results already exist. For example, use
`AF2_RELAX_VERSIONS="v3"` to relax only v3, or explicitly include `v1` when
reproducing the legacy workflow. Custom inputs outside the standard
`*.colabfold.v1/`, `.v2/` and `.v3/` directory layout remain eligible and are
controlled through `INPUT_DIR` and `PDB_PATTERN`.

With the chapter-compatible installation from
[section 2.5](../2.5_Greasy/2.5_README.md), configure Greasy as:

```bash
export GREASY_HOME="/opt/greasy/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"
```

### AF2 relaxation examples

Relax the default v2 and v3 models in all extracted case-study AF2 folders:

```bash
export GREASY_HOME="/opt/greasy/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"
export GREASY_NWORKERS=4

bash relax_alphafold2_models.sh
```

Relax only v3 models:

```bash
export AF2_RELAX_VERSIONS="v3"

bash relax_alphafold2_models.sh
```

Relax only one case:

```bash
export CASE="4POU"
export GREASY_HOME="/opt/greasy/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"

bash relax_alphafold2_models.sh
```

Only create the Greasy task file:

```bash
export CASE="4POU"
export RUN_GREASY=0

bash relax_alphafold2_models.sh
```

With `RUN_GREASY=1`, `colabfold_relax` and `${GREASY}/greasy` must both be
executable before Greasy is launched. With `RUN_GREASY=0`, a missing
`colabfold_relax` produces a warning but does not prevent task-file generation;
the command must be available in the environment where that task file is later
executed.

The task file contains commands equivalent to:

```bash
timeout 8m colabfold_relax --max-iterations 2000 --tolerance 2.39 \
  --stiffness 10.0 --max-outer-iterations 3 --use-gpu input.pdb output.pdb
```

### GPU memory and relaxation timeout

Each Greasy worker can launch an independent GPU relaxation. When several
workers use the same GPU, their memory allocations compete for the available
VRAM and can produce an error such as:

```text
jaxlib.xla_extension.XlaRuntimeError:
RESOURCE_EXHAUSTED: CUDA_ERROR_OUT_OF_MEMORY: out of memory
```

The frequency of this failure depends on both the size of the protein complex
and the GPU VRAM. Even when most models complete successfully, a small number
may still fail because GPU-memory requirements vary between structures. The
default `GREASY_NWORKERS=4` is therefore not universally safe. Start with
`GREASY_NWORKERS=1`, test representative models, and increase the value
gradually rather than selecting a high worker count from the beginning. If an
out-of-memory error occurs, reduce the number of workers; the relaxation
timeout does not prevent VRAM exhaustion.

Every generated relaxation task is limited by `RELAX_TIMEOUT`, which defaults
to eight minutes. This prevents an unusually slow or severely clashing model
from occupying a Greasy worker indefinitely and allows the remaining queue to
continue. A timed-out output, if one was created, must not be treated as a
completed relaxed model. Inspect it before regenerating the selected task with
`OVERWRITE=1` and, when justified, a longer `RELAX_TIMEOUT`.

> **Restarting an incomplete Greasy batch**
>
> When tasks fail or remain unfinished, Greasy can create a restart file by
> appending `.rst` to the original task-file name, for example
> `4POU.af2_relax.greasy.rst`. Inspect `greasy.log` and the restart file before
> submitting it. For failures caused by GPU-memory exhaustion, reduce the
> number of workers and run the remaining tasks from the directory containing
> the restart file:
>
> ```bash
> GREASY_NWORKERS=1 "${GREASY}/greasy" 4POU.af2_relax.greasy.rst
> ```
>
> This avoids submitting the complete original task list again. Address the
> cause of the failure before restarting: reducing `GREASY_NWORKERS` can resolve
> VRAM exhaustion, whereas a task that reached `timeout 8m` will retain that
> same limit in the `.rst` file. To change its timeout, regenerate the selected
> task with a different `RELAX_TIMEOUT`.

## Prepared inputs for AlphaFold Server

`json_files_for_alphafold3_server/` contains three syntactically valid JSON
input files and no subdirectories or result files:

| File | Prepared protein composition | Seed | Notes |
| --- | --- | --- | --- |
| `1WEJ_alphafoldserver.json` | 3 protein entries (214, 223 and 105 residues) | `13441` | One copy of each sequence. |
| `2FJG_alphafoldserver.json` | 4 protein entries (218, 211, 95 and 95 residues) | `13441` | The third and fourth entries are identical sequences, each with `count: 1`. |
| `4POU_alphafoldserver.json` | 2 protein entries (121 and 124 residues) | `13441` | One copy of each sequence; this is the default example used by the local scripts. |

Each file defines one protein-only AlphaFold Server job with the case name,
sequences, copy counts and model seed already prepared. The jobs do not include
ligands, DNA, RNA, modifications, templates or custom MSAs.

### Using the server JSON files

1. Select the JSON matching the case and inspect its `name`, sequence order,
   sequence copy count and seed.
2. Import or upload that file manually in the AlphaFold Server interface and
   review the parsed job before submitting it.
3. Run the prediction on the server and download the resulting models and
   confidence files. These are external calculation steps and are not performed
   by any script in this repository.
4. Extract each downloaded server result under
   `4_Case_Studies/<CASE>/AF3/fold*/`, preserving a distinct `fold*`
   directory for each server job. The section 3.3.2 workflow prepares the PDB
   models required for pyDock from the downloaded ranked CIF files.

The `AF3/output/<CASE>/` hierarchy is reserved for predictions produced by the
local `run_alphafold3.sh` workflow; it is not the destination for AlphaFold
Server downloads.

The prepared JSON files avoid re-entering the sequences, stoichiometry and seed,
but they do not contain precomputed structures. Precomputed AF2 and AF3 outputs
for the `2FJG` and `4POU` case studies are supplied separately in the tracked
archives `4_Case_Studies/2FJG/2FJG.tar.gz` and
`4_Case_Studies/4POU/4POU.tar.gz`; extract those archives when the goal is
to test downstream steps without rerunning AlphaFold. No corresponding `1WEJ`
results archive is present in this repository.

> **Practical checks:** the server JSON files do not assign explicit chain IDs.
> Verify the entity order, repeated copies (especially the duplicated 95-residue
> sequence in `2FJG`) and the chain IDs in downloaded structures before pyDock
> setup. Also confirm whether the biological system requires ligands, modified
> residues or other entities omitted from these protein-only examples. Change
> seeds or job definitions only deliberately and keep a record of the submitted
> JSON for reproducibility.

> **CIF/PDB note:** keep the downloaded CIF as the canonical server result.
> PDB has stricter chain, residue and atom-number limits, and `cif_to_pdb.py`
> may add `SSBOND`, `LINK`, `CONECT` and `MODRES` records. Inspect converted
> chains, ligands, modifications and connections before pyDock scoring.

AlphaFold Server and local AlphaFold 3 use different JSON formats. Upload the
prepared server files through the web interface; for local predictions, let
`run_alphafold3.sh` create `${CASE}.json`. Do not pass a server JSON directly
to the local runner without adapting and validating it first.

## Script 3: `run_alphafold3.sh`

This script prepares and runs a local AlphaFold 3 job. It converts `SEQ` into
an AlphaFold 3 JSON request, assigning chain IDs automatically from `A` to `Z`,
then calls the configured AlphaFold 3 runner. After AF3 finishes, the script
converts sample CIF models to PDB files with `cif_to_pdb.py`.

The automatic JSON generator currently supports protein sequences only. It does
not generate ligand, DNA, RNA, glycan or custom modification entries.

### AF3 input variables

| Variable | Default | Purpose |
| --- | --- | --- |
| `CASE` | `4POU` | Case identifier used for folders and file names. |
| `SEQ` | 4POU VHH/RNase A sequence | Protein sequences. Separate chains with `:`. |
| `MODEL_SEEDS` | `13441` | One seed or several comma-separated integer seeds. |
| `APPDIR` | `/home/user/Programs` | Base software directory. |
| `ALPHAFOLD3DIR` | `${APPDIR}/alphafold3` | AlphaFold 3 installation directory. |
| `AF3_CONDA_ENV` | `Alphafold3` | Expected Conda environment name. A non-Conda installation can also be used. |
| `HMMER3_BINDIR` | empty | Optional explicit directory containing all HMMER executables. Otherwise use `${CONDA_PREFIX}/bin`, or search the shell `PATH` when no Conda environment is active. |
| `DB_DIR` | `${ALPHAFOLD3DIR}/public_databases` | AlphaFold 3 database directory. |
| `MODEL_DIR` | `${ALPHAFOLD3DIR}/models` | AlphaFold 3 model parameter directory. |
| `RUN_AF3` | `run_alphafold.py` | AlphaFold 3 runner executable name or explicit executable path. |
| `BUCKETS` | AF3 default bucket list | Bucket sizes passed to AlphaFold 3. |
| `CIF_TO_PDB` | `cif_to_pdb.py` next to `run_alphafold3.sh` | CIF-to-PDB conversion helper. |

Before launching AlphaFold 3, make sure that the AF3 runner, all five HMMER
commands, the database directory and the model-parameter directory are
available. Set `HMMER3_BINDIR` when the HMMER commands are not already
available through the selected environment or `PATH`.

### AF3 usage example

From this section directory:

```bash
conda activate Alphafold3

export CASE="MyComplex"
export SEQ="AAAAABBBB:CCCCDDDD"
export MODEL_SEEDS="13441,2024"
export ALPHAFOLD3DIR="/path/to/alphafold3"
export HMMER3_BINDIR="/path/to/conda/env/bin"
export DB_DIR="/path/to/alphafold3/public_databases"
export MODEL_DIR="/path/to/alphafold3/models"
export RUN_AF3="run_alphafold.py"

bash run_alphafold3.sh
```

From the repository root, invoke the same script as:

```bash
CASE="MyComplex" SEQ="AAAAABBBB:CCCCDDDD" \
  bash 3.1_Generating_3D_Models_for_Protein_Protein_Complexes_with_AlphaFold/run_alphafold3.sh
```

The local AF3 results are written to:

```text

    └── AF3/
        ├── ${CASE}.json
        └── output/
            ├── af3_run.log
            └── ${CASE}/

}
```

## Requirements

AF2 workflow:

* ColabFold: `colabfold_batch`
* CUDA-capable GPU compatible with the ColabFold/OpenMM relaxation
* GNU Bash

Optional AF2 relaxation:

* ColabFold: `colabfold_relax`
* [Greasy job scheduler](../2.5_Greasy/2.5_README.md)
* GNU Coreutils: `timeout`
* CUDA-capable GPU
* GNU Bash

AF3 workflow:

* Local AlphaFold 3 installation
* AlphaFold 3 databases and model parameters
* HMMER executables: `jackhmmer`, `nhmmer`, `hmmalign`, `hmmsearch`,
  `hmmbuild`
* Biopython for `cif_to_pdb.py`
* Python available in the active environment
* GNU Bash

The root [`environment.yml`](../environment.yml) provides Python and Biopython
for standalone CIF-to-PDB conversion and for the analysis notebook. It does not
install the local AlphaFold3 runtime, databases or model parameters; continue
to use the dedicated AF3 installation for local predictions.

## Troubleshooting

| Issue | Likely cause |
| --- | --- |
| Expected Conda environment warning | The default `Alphafold2` or `Alphafold3` environment is not active. The script can continue if all required executables are resolved. |
| `colabfold_batch` not found | ColabFold is not installed or not active in `PATH`. |
| `colabfold_relax` not found | ColabFold relaxation tools are not installed or not active in `PATH`. |
| No GPU detected | CUDA is not installed, not visible or not available to the job. |
| Greasy does not start | Check that `GREASY_HOME` is the installation prefix and that `GREASY` points to the wrapper directory, normally `${GREASY_HOME}/bin`. |
| AF2 relaxation worker count does not change | Set `GREASY_NWORKERS` before launching the script. |
| AF2 relaxation fails with `RESOURCE_EXHAUSTED` or `CUDA_ERROR_OUT_OF_MEMORY` | Too many concurrent relaxation tasks are competing for the available GPU VRAM. Reduce `GREASY_NWORKERS` and test again. |
| AF2 relaxation exits after eight minutes | The default `RELAX_TIMEOUT=8m` was reached. Inspect the model and any partial output before retrying with `OVERWRITE=1` and a longer timeout. |
| AF3 reports missing HMMER binary | `HMMER3_BINDIR` does not point to the active environment or system HMMER directory. |
| AF3 reports missing databases or models | `DB_DIR` or `MODEL_DIR` does not point to a valid AlphaFold 3 installation. |
| AF3 JSON generation fails | `SEQ` is empty, has more than 26 chains, or contains unsupported characters. |
| AF3 Server conversion fails | Check that Python and Biopython are available and that the downloaded ranked model CIF is stored under the case `AF3/fold*/` directory. |
