# Greasy

This directory documents the Greasy installation and usage associated with
section 2.5 of the *Methods in Molecular Biology* chapter. Greasy executes a
text file containing independent shell commands and keeps the available
workers occupied as tasks finish. In this repository it is used to parallelize
AlphaFold2 model relaxation and pyDock binding-energy calculations.

Greasy was developed by the HPC User Support Team at the Barcelona
Supercomputing Center (BSC). This repository does not redistribute or modify
the Greasy source code; it downloads the selected source archive directly from
the official project.

## Version and directory naming

The commands below install official source tag `2.1.2.1` in a directory named
`GREASY_2.2`, following the naming used in the chapter:

```text
<installation_parent>/GREASY_2.2/
```

The directory name and source version are therefore intentionally different.
If you use a later official version, install it under a separate, clearly
named prefix and adjust the paths shown below.

## Requirements

Greasy is built on Linux using standard compilation tools. `wget` is used by
the commands below only to retrieve the source archive.

On Debian or Ubuntu:

```bash
sudo apt-get update
sudo apt-get install build-essential wget
```

On Red Hat or CentOS-compatible distributions:

```bash
sudo yum groupinstall "Development Tools"
sudo yum install wget
```

On distributions that use `dnf`, use the equivalent `dnf group install` and
`dnf install` commands. Autoconf and Automake are additionally needed when
building directly from a development checkout whose generated build files
need to be recreated.

Slurm, SSH and MPI support are optional and depend on the environment in which
Greasy will run. The repository workflows described below only require a
working local Greasy installation.

## Installation in a user-owned directory

Installing without administrator privileges is recommended when a shared
system-wide installation is not required. The following keeps the chapter's
`GREASY_2.2` directory name:

```bash
GREASY_VERSION="2.1.2.1"
GREASY_PARENT="${HOME}/software"
GREASY_HOME="${GREASY_PARENT}/GREASY_2.2"

mkdir -p "${GREASY_PARENT}"
cd "${GREASY_PARENT}"

wget -O "${GREASY_VERSION}.tar.gz" \
  "https://github.com/BSC-Support-Team/GREASY/archive/refs/tags/${GREASY_VERSION}.tar.gz"
tar -xzf "${GREASY_VERSION}.tar.gz"
cd "GREASY-${GREASY_VERSION}"

./configure --prefix="${GREASY_HOME}"
make
make install
```

After installation, the Greasy command used by this repository is:

```text
<installation_parent>/GREASY_2.2/bin/greasy
```

Install directly at the final desired path. Moving the completed directory is
not recommended because the generated wrapper and configuration contain paths
derived from the `--prefix` value.

## Chapter-style installation under `/opt`

If `/opt/greasy` is the intended installation parent, create it first and give
the installing user ownership. Replace the user and group handling if required
by the local administrator:

```bash
sudo mkdir -p /opt/greasy
sudo chown -R "${USER}:$(id -gn)" /opt/greasy

GREASY_PARENT="/opt/greasy"
GREASY_HOME="${GREASY_PARENT}/GREASY_2.2"
```

Then use the same download and build commands from the previous section. The
correct configuration command remains a single command:

```bash
./configure --prefix="${GREASY_HOME}"
```

## Environment used by this repository

A standard installation places the public wrapper at
`<prefix>/bin/greasy`. The scripts in sections 3.1 and 3.3.2 accept
`GREASY_HOME` as that installation prefix and derive `GREASY` as the directory
containing the wrapper. Configure the environment as follows:

```bash
export GREASY_HOME="${HOME}/software/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"
export PATH="${GREASY}:${PATH}"
```

For the `/opt` example:

```bash
export GREASY_HOME="/opt/greasy/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"
export PATH="${GREASY}:${PATH}"
```

These exports can be added to `~/.bashrc` after confirming the installation
path. `GREASY_HOME` identifies the installation prefix and `GREASY` identifies
the directory containing the `greasy` command. The repository scripts execute:

```bash
"${GREASY}/greasy" <task-file>
```

## Verification

First verify that the wrapper exists and is executable:

```bash
test -x "${GREASY}/greasy"
```

A lightweight smoke test can then execute two independent commands:

```bash
printf '%s\n' '/bin/echo task-one' '/bin/echo task-two' \
  > /tmp/greasy_smoke_test.txt

GREASY_NWORKERS=2 \
GREASY_LOGFILE=/tmp/greasy_smoke_test.log \
  "${GREASY}/greasy" /tmp/greasy_smoke_test.txt
```

The final log summary should report two successful tasks. This test does not
run AlphaFold, relaxation or pyDock.

## Task files used in this repository

For the repository workflows, each non-comment line is one independent shell
command.

### AlphaFold2 relaxation

`relax_alphafold2_models.sh` writes a task file such as:

```text
3.1_Generating_3D_Models_for_Protein_Protein_Complexes_with_AlphaFold/
└── 4POU.af2_relax.greasy
```

Each task runs `colabfold_relax` for one PDB model. From the repository root:

```bash
export GREASY_HOME="${HOME}/software/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"

CASE="4POU" \
GREASY_NWORKERS=4 \
  bash 3.1_Generating_3D_Models_for_Protein_Protein_Complexes_with_AlphaFold/relax_alphafold2_models.sh
```

Set `RUN_GREASY=0` to create the task file without launching Greasy.

### pyDock binding-energy calculations

`ini_creator_bindE.sh` writes the combined task list to:

```text
4_Case_Studies/<CASE>/Greasy_BindE_mul_ligs.txt
```

Each task changes to the relevant AF2 or AF3 model directory and runs one
`pyDock3 ... bindEy` calculation. From the repository root:

```bash
export GREASY_HOME="${HOME}/software/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"

CASE="4POU" \
PYDOCK="/path/to/pyDock3" \
GREASY_NWORKERS=8 \
  bash 3.3.2_Computing_pyDock_Scores_for_a_Set_of_Complexes/ini_creator_bindE.sh
```

The number of workers should not exceed the resources assigned to the run.
This is particularly important when several relaxation commands compete for
the same GPU.

## Logs and runtime configuration

The installed configuration is stored in:

```text
${GREASY_HOME}/etc/greasy.conf
```

Environment variables override the corresponding configuration values. The
most relevant variables for this repository are:

| Variable | Purpose |
| --- | --- |
| `GREASY_NWORKERS` | Number of tasks that can run concurrently. |
| `GREASY_LOGFILE` | Explicit output path for the Greasy log. |
| `GREASY_LOGLEVEL` | Log verbosity; level 3 is the normal information level. |
| `GREASY_MAXRETRIES` | Number of retries for failed tasks. |
| `GREASY_ENGINE` | Runtime engine, normally `basic` for local use. |

The default upstream configuration uses the relative filename `greasy.log`,
which is created in the directory from which Greasy is launched. Set an
explicit path when the log must be kept with a case study, for example:

```bash
export GREASY_LOGFILE="/path/to/AF_pyDock/4_Case_Studies/4POU/greasy_bindE.log"
```

If tasks fail or remain unfinished, Greasy can produce a restart task file
with the `.rst` suffix. Inspect the main log and this file, then submit the
remaining tasks with the same Greasy installation, for example:

```bash
"${GREASY}/greasy" 4POU.af2_relax.greasy.rst
```

## Troubleshooting

| Problem | Check |
| --- | --- |
| `.../greasy: No such file or directory` | `GREASY` must point to `GREASY_2.2/bin`, not only to `GREASY_2.2`. |
| Greasy was moved after installation and no longer starts | Reinstall it at its final path. |
| No tasks run in parallel | Set `GREASY_NWORKERS` to a positive value appropriate for the allocation. |
| `greasy.log` appears in an unexpected directory | Set an absolute `GREASY_LOGFILE`. |
| Tasks fail although Greasy starts | Inspect the external command in the task file and the Greasy log. |
| Permission denied during installation | Use a user-owned prefix or ask the system administrator to prepare `/opt/greasy`. |
