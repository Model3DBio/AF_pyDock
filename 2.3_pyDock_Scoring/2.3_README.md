# pyDock scoring

This directory documents the pyDock installation and scoring functionality
associated with section 2.3 of the *Methods in Molecular Biology* chapter.
[pyDock](https://life.bsc.es/pid/pydock/) is a protein–protein docking suite
developed by the Protein Interactions and Docking Group, Life Sciences
Department, at the [Barcelona Supercomputing Center
(BSC)](https://www.bsc.es/).

The complete pyDock3 distribution provides modules for preparing structures,
generating or processing docking poses, calculating energies and analysing
results. The workflow in this repository uses only its scoring functionality:
pre-existing AlphaFold2 or AlphaFold3 complex models are evaluated with the
`bindEy` module. It does not use pyDock to sample new docking poses.

## Version and system compatibility

The chapter and the current repository workflow target **pyDock3 version
3.2.3**. Its distributed Linux executable is a 32-bit binary. A 64-bit Linux
system therefore needs the corresponding 32-bit runtime compatibility
libraries.

On Debian or Ubuntu-compatible systems, the pyDock documentation gives:

```bash
sudo dpkg --add-architecture i386
sudo apt-get update
sudo apt-get install zlib1g:i386
```

On Red Hat or CentOS-compatible systems:

```bash
sudo yum install glibc.i686 zlib.i686
```

Package names may differ on newer distributions. Install compatibility
packages only through the system's normal package manager; a pyDock loader
failure should not be worked around by copying arbitrary shared libraries into
the distribution directory.

## Installation

After receiving the licensed distribution, extract it at the final location.
The exact archive name supplied by the authors may vary:

```bash
mkdir -p "${HOME}/software"
cd "${HOME}/software"
tar -xzf /path/to/pydock-distribution.tar.gz
cd pyDock3
chmod go+rx data
```

If the extracted directory has a different name, use that real path in the
environment configuration below. A typical pyDock3 distribution contains:

```text
pyDock3/
├── data/                 # data required by pyDock
├── etc/
│   └── pydock.conf       # paths and pyDock configuration
├── parallel/             # helper scripts for parallel workflows
├── LICENSE
├── README
├── pyDock3               # executable
└── scwrl3_lin.tar.gz     # SCWRL3 archive bundled with pyDock3
```

Some distributions may place an additional configuration file at the top
level. Follow their bundled `README` and update any external-program paths in
the configuration actually used by that distribution. Installation of the
bundled SCWRL3 archive is documented separately in [section
2.4](../2.4_SCWRL/2.4_README.md).

## Environment configuration

Set `PYDOCK` to the directory that contains the `pyDock3` executable:

```bash
export PYDOCK="${HOME}/software/pyDock3"
```

The scripts in this repository call `"${PYDOCK}/pyDock3"` directly, so adding
pyDock to `PATH` is not required for their normal use. It is convenient for
interactive use and for the helper scripts distributed with pyDock:

```bash
export PATH="${PYDOCK}:${PYDOCK}/parallel:${PATH}"
```

Add these exports to `~/.bashrc` only after verifying the installation path.
Additional programs such as FTDock or ZDOCK are needed for some full docking
modules, but are not required by the scoring-only workflow described here.

## Verification

First check that the executable is present:

```bash
test -x "${PYDOCK}/pyDock3"
```

Then run it without arguments as a lightweight loader test:

```bash
"${PYDOCK}/pyDock3"
```

For version 3.2.3, seeing the pyDock banner, version and usage information
followed by a `wrong parameters` error is expected: no docking job was
specified. If the banner does not appear, check the executable permissions and
32-bit compatibility libraries before configuring this repository's batch
workflow.

## Scoring interface used by this repository

The pyDock command used for each model is:

```bash
"${PYDOCK}/pyDock3" <ini-base-name> bindEy
```

`<ini-base-name>` is passed without the `.ini` suffix. The INI file identifies
the input PDB and assigns its chains to receptor and ligand, for example:

```ini
[receptor]
pdb = model.pdb
mol = A
newmol = A

[ligand]
pdb = model.pdb
mol = B
newmol = B
```

The principal result is an energy table with the same base name and an `.ene`
suffix. Its standard columns are:

```text
Conf  Ele  Desolv  VDW  Total  RANK
```

`Ele`, `Desolv` and `VDW` are the electrostatic, desolvation and van der Waals
components. `Total` is the native pyDock score, which uses the weighted van der
Waals contribution (`Ele + Desolv + 0.1 × VDW`). In the chapter and in the
derived tables of this repository, that score is called `pyDock-0.1VDW`; the
full-VDW score recalculated as `Ele + Desolv + VDW` is called `pyDock-1VDW`.
The native `.ene` headers are not changed.

pyDock may also create prepared receptor or ligand PDB files as auxiliary
output. Downstream analysis reads the `.ene` components and calculates
rankings across the complete set of AF2 and AF3 models; the rank recorded
inside an individual energy file is not treated as that global rank.

## Repository workflow

[`ini_creator_bindE.sh`](../3.3.2_Computing_pyDock_Scores_for_a_Set_of_Complexes/ini_creator_bindE.sh)
generates the INI files beside the selected AF2 and AF3 PDB models and writes
one `bindEy` command per model to a Greasy task file. The associated [section
3.3.2 guide](../3.3.2_Computing_pyDock_Scores_for_a_Set_of_Complexes/3.3.2_README.md)
documents model discovery, receptor/ligand chain selection and output paths.

For example, from the repository root:

```bash
export PYDOCK="${HOME}/software/pyDock3"
export GREASY="${HOME}/software/GREASY_2.2/bin"

CASE="4POU" \
CHAINS_REC_LIG_VALUES="A B" \
  bash 3.3.2_Computing_pyDock_Scores_for_a_Set_of_Complexes/ini_creator_bindE.sh
```

This command runs pyDock calculations through Greasy. Review the generated
INI files and chain assignments before launching a large model set. Greasy
installation, worker configuration and task logs are covered in [section
2.5](../2.5_Greasy/2.5_README.md).
