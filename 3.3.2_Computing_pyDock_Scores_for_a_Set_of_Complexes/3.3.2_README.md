# Batch pyDock3 scoring for AlphaFold models

This directory contains `ini_creator_bindE.sh`, which prepares pyDock3
configuration files (`.ini`) for AlphaFold2 and AlphaFold3 models and writes a
Greasy task list for calculating their binding energies. When an experimental
complex is available, the same workflow can optionally calculate reference-based
CAPRI metrics. One or more receptor–ligand configurations can be evaluated for
each model. If an input structure has incomplete side chains, pyDock3 can use
the SCWRL3 executable configured in its own `etc/pydock.conf`.

---

## Workflow summary

```
AF2/AF3 Models → Chain Definitions → pyDock INI Files → Greasy Job List → Binding-Energy Calculations
                                                                           └→ Optional CAPRI Assessment
```

`ini_creator_bindE.sh` processes one case per invocation and can be launched
from the repository root or this section directory. It reads the selected
models and writes the resulting pyDock files under
`4_Case_Studies/<CASE>/`. The accepted model locations and names are:

* `4_Case_Studies/<CASE>/AF2/*cola*v*/` for AF2 `*[0-9].pdb` files;
* `4_Case_Studies/<CASE>/AF3/fold*/` for AlphaFold Server models;
* `4_Case_Studies/<CASE>/AF3/output/<CASE>/seed-*/` for local AF3
  `*_sample-[0-9]*_model.pdb` files.

At least one of the case `AF2/` or `AF3/` directories must exist. The available
prediction branches can be processed separately or together. AlphaFold Server
downloads may contain ranked CIF models; the script prepares the corresponding
PDB files before creating the pyDock tasks. Downloaded template CIF files are
not treated as prediction models. If a model cannot be converted, correct that
input before running the scoring batch.

## Preparing the supplied case-study inputs

The script needs the case-study prediction hierarchy described above. There are
two ways to populate it:

1. Generate the predictions locally with the workflows in
   [section 3.1](../3.1_Generating_3D_Models_for_Protein_Protein_Complexes_with_AlphaFold/3.1_README.md).
2. Use a supplied precomputed case archive to avoid rerunning the
   computationally expensive AlphaFold calculations.

In a clean clone, the supplied cases remain compressed. If local AF2 and AF3
predictions will not be generated, extract the selected archive inside its case
directory before continuing. For the included 2FJG and 4POU cases:

```bash
CASE="4POU"  # or 2FJG
tar -xzf "4_Case_Studies/${CASE}/${CASE}.tar.gz" \
  -C "4_Case_Studies/${CASE}"
```

This extraction step is unnecessary when the corresponding `AF2/` and `AF3/`
directories have already been generated or extracted. Avoid combining an
archive with an existing model set unless that merge is intentional.

> [!IMPORTANT]
> **Relax the AF2 recycle models before the recommended pyDock analysis.**
> The supplied archives contain the saved AF2 recycle structures in unrelaxed
> form, together with the standard relaxed final models generated by the
> original ColabFold runs. They do not contain a complete relaxed recycle
> ensemble. Because the downstream workflow recommends scoring relaxed AF2
> structures, run `relax_alphafold2_models.sh` after extraction and before
> calculating the pyDock energies. AF3 models are kept and scored as predicted.

After configuring ColabFold and Greasy as described in section 3.1, the default
AlphaFold2-Multimer v2 and v3 recycle models for the selected case can be
relaxed from the repository root with:

```bash
CASE="${CASE}" AF2_RELAX_VERSIONS="v2 v3" \
  bash 3.1_Generating_3D_Models_for_Protein_Protein_Complexes_with_AlphaFold/relax_alphafold2_models.sh
```

Include `v1` in `AF2_RELAX_VERSIONS` only when the legacy
AlphaFold2-Multimer v1 models are also required. The relaxed PDB files are
written beside their unrelaxed source models.

The archives provide precomputed AlphaFold model-generation outputs, but they
do not include pyDock `.ene` tables. The scoring workflow in this section must
therefore still be run after preparing the desired AF2 and AF3 model sets.

> **Practical note:** `ini_creator_bindE.sh` discovers both relaxed and
> unrelaxed AF2 PDB files when both are present in the prediction directories.
> It will therefore generate pyDock tasks for both sets. The section 3.4
> notebook excludes unrelaxed AF2 models by default from the recommended
> combined analysis.

## Input variables

The script uses several environment variables, each with a default value.
Override them as needed before execution.

---

### **1. `CASE` — Case identifier**

**Purpose:**
Selects the case to search
(`<repo_root>/4_Case_Studies/${CASE}/`).

**Default:**

```bash
4POU
```

**Override:**

```bash
export CASE="MyProject"
```

---

### **2. `PYDOCK` and `PYDOCK_BINARY` — pyDock executable**

**Purpose:**
`PYDOCK` specifies the installation directory and `PYDOCK_BINARY` the name of
the executable called from that directory.

**Default:**

```
PYDOCK=/usr/local/software/pyDock3/
PYDOCK_BINARY=pyDock3
```

**Override:**

```bash
export PYDOCK="/opt/pydock3"
export PYDOCK_BINARY="pydock3"
```

`PYDOCK_BINARY` can be set to `pyDock3`, `pyDock4`, `pydock3` or `pydock4`
as appropriate.

Installation, license, compatibility and verification instructions are in
[section 2.3](../2.3_pyDock_Scoring/2.3_README.md).

### SCWRL3 used internally by pyDock3

`ini_creator_bindE.sh` does not require a separate `SCWRL` environment
variable and does not call SCWRL itself. When side-chain reconstruction is
needed, pyDock3 reads the executable path from its configuration file:

```ini
# $PYDOCK/etc/pydock.conf
[apps]
SCWRL=/absolute/path/to/pyDock3/3.0/scwrl3
```

The pyDock3 distribution used by this workflow includes the SCWRL3 archive.
Follow [section 2.4](../2.4_SCWRL/2.4_README.md) to install that bundled copy
and configure its absolute path. SCWRL4 should not be substituted in this
configuration unless the installed pyDock version explicitly supports it.

---

### **3. `GREASY_HOME` and `GREASY` — Greasy installation paths**

**Purpose:**
Defines the Greasy installation prefix and the directory containing the public
`greasy` wrapper.

**Default:**

```bash
GREASY_HOME=/path/to/software/GREASY_2.2
GREASY=${GREASY_HOME}/bin
```

**Override:**

```bash
export GREASY_HOME="/opt/greasy/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"
```

This convention matches the installation documented in
[section 2.5](../2.5_Greasy/2.5_README.md). `ini_creator_bindE.sh` invokes
`${GREASY}/greasy` directly. For a nonstandard installation, set `GREASY` to
the directory containing that wrapper.

---

### **4. `GREASY_NWORKERS` — Number of parallel workers**

**Default:**

```bash
8
```

Set based on CPU/GPU availability:

```bash
export GREASY_NWORKERS=4
```

---

### **5. `CHAINS_REC_LIG_VALUES` — Receptor/ligand chain definitions**

**Purpose:**
Controls how chains inside each PDB model are assigned to **receptor** and **ligand** when generating `INI` files for pyDock3.

Each entry in this array must follow:

```
"RECEPTOR_CHAINS LIGAND_CHAINS"
```

#### Interpretation rules

* Text **before** the space → receptor chain(s)
* Text **after** the space → ligand chain(s)
* Multiple chains are allowed via commas (`A,B`)
* **Each array entry produces one `INI` file per PDB model**
* File names encode both assignments as
  `*_LIG_<LIGAND>_REC_<RECEPTOR>.ini`; commas are represented by hyphens

If you define **N entries**, the script will generate **N `INI` files per model**.
Configurations that normalize to the same filename suffix are rejected before
any task list is written.

---

## Chain definition examples

### **1. Simple case**

```bash
CHAINS_REC_LIG_VALUES=("A B" "C B")
source ./ini_creator_bindE.sh
```

Generates **two** `INI` files for every PDB model:

* Receptor = A, Ligand = B → `*_LIG_B_REC_A.ini`
* Receptor = C, Ligand = B → `*_LIG_B_REC_C.ini`

Including the receptor prevents these two configurations with the same ligand
from overwriting each other.

Multiple values are a Bash array and are therefore used by sourcing the script
from this section directory; Bash arrays are not inherited by a child process
started with `bash ini_creator_bindE.sh`. A single definition can be exported
normally, as shown below.

---

### **2. Multiple receptor chains**

```bash
export CHAINS_REC_LIG_VALUES="A,B C"
```

Interpretation:

* Receptor = A,B
* Ligand = C

Generates **one** `INI` file per PDB:

```
*_LIG_C_REC_A-B.ini
```

---

### Summary table

| Definition             | Receptor | Ligand | File suffix                    |
| ---------------------- | -------- | ------ | ------------------------------ |
| `"A B"`                | A        | B      | `_LIG_B_REC_A.ini`             |
| `"C B"`                | C        | B      | `_LIG_B_REC_C.ini`             |
| `"A,B C"`              | A,B      | C      | `_LIG_C_REC_A-B.ini`           |
| Multiple array entries | —        | —      | One distinct suffix per entry  |

> **Compatibility with section 3.4:** this script can generate several
> receptor–ligand configurations for one model, but the current
> `Analysis_protocol.ipynb` workflow accepts only one pyDock `.ene` file per
> primary model. If multiple configurations for the same model are present, the
> notebook reports the ambiguous mapping and stops. Use one consistent
> receptor–ligand configuration for the combined analysis, or keep alternative
> configurations in separate analysis inputs.

> **Before rescoring an existing model directory:** outputs named only with
> `_LIG_<LIGAND>` remain readable by the notebook, but they must not coexist
> with `_LIG_<LIGAND>_REC_<RECEPTOR>` outputs for the same PDB. Before
> recalculating, move one result set to a separate archive directory; the
> script does not alter existing outputs automatically.

### **6. Optional reference-based CAPRI assessment**

The standard energy-scoring workflow remains reference-free. Enable the
optional comparison only when an appropriate experimental complex is
available:

```bash
export RUN_CAPRI_RMSD=1
export REFERENCE_PDB="AF2/4POU.colabfold.v2/4POU.pdb"
export CHAINS_REC_LIG_VALUES="A B"
export REFERENCE_CHAINS_REC_LIG_VALUES="B A"
```

`REFERENCE_PDB` may be absolute or relative to
`4_Case_Studies/<CASE>/`. The relative path above therefore resolves to the
reference structure supplied inside the extracted 4POU case.

Each `REFERENCE_CHAINS_REC_LIG_VALUES` entry uses the same two-field form as
the model definitions:

```text
"REFERENCE_RECEPTOR_CHAINS REFERENCE_LIGAND_CHAINS"
```

Entries are matched by position with `CHAINS_REC_LIG_VALUES`. The two arrays
must have the same length. For each position, the model receptor and reference
receptor must contain the same number of comma-separated chains; the equivalent
rule is checked separately for the two ligands. Their order defines the chain
correspondence. In the 4POU example, model receptor A corresponds to reference
chain B, while model ligand B corresponds to reference chain A. The resulting
INI section is:

```ini
[reference]
pdb = <absolute-path-to-4POU.pdb>
recmol = B
ligmol = A
newrecmol = A
newligmol = B
```

For several receptor–ligand configurations, define both variables as indexed
Bash arrays in the same order and source the script, as described for
`CHAINS_REC_LIG_VALUES`. Associative arrays are not accepted. Before converting
Server models or replacing the Greasy task list, the script checks that the
reference file is nonempty and that the definitions have the expected form,
number and chain-list cardinality. pyDock later checks whether the named chains
can be processed; the user remains responsible for choosing the biologically
correct correspondence.

Leave `RUN_CAPRI_RMSD=0` (the default) and do not define reference variables to
generate the original binding-energy workflow. Supplying reference variables
without enabling the option is treated as an error rather than being silently
ignored.

## Output structure

After execution, the script produces:

```
4_Case_Studies/${CASE}/
├── Greasy_BindE_mul_ligs.txt         # Combined Greasy command list
├── AF2/
│   └── *colab*v*/
│       ├── *[0-9].pdb
│       ├── generated *_LIG_*_REC_*.ini
│       ├── generated *_LIG_*_REC_*.ene
│       ├── generated *_LIG_*_REC_*_rec.pdb
│       └── generated *_LIG_*_REC_*_lig.pdb
└── AF3/
    ├── fold*/                        # AlphaFold Server output
    │   ├── *_model_<rank>.cif        # Downloaded ranked model
    │   ├── *_model_<rank>.pdb        # Generated automatically from CIF
    │   ├── generated *_LIG_*_REC_*.ini
    │   ├── generated *_LIG_*_REC_*.ene
    │   ├── generated *_LIG_*_REC_*_rec.pdb
    │   └── generated *_LIG_*_REC_*_lig.pdb
    └── output/${CASE}/seed-*/        # Local AlphaFold 3 output
        ├── *_sample-[0-9]*_model.pdb
        ├── generated *_LIG_*_REC_*.ini
        ├── generated *_LIG_*_REC_*.ene
        ├── generated *_LIG_*_REC_*_rec.pdb
        └── generated *_LIG_*_REC_*_lig.pdb
```

Additional files produced beside each model configuration are:

```text
<configuration>.rot                 # bindEy auxiliary
<configuration>_ref.pdb             # with an enabled reference
<configuration>.RMSD.capri          # optional capriRMSD result
```

The shell script creates the `.ini` files and the combined Greasy task list.
Each successful pyDock `bindEy` task then creates the `.ene` energy table and
may create parsed receptor and ligand structures ending in `_rec.pdb` and
`_lig.pdb`. These PDB files are pyDock auxiliaries, not additional AlphaFold
models; the section 3.4 notebook excludes them from the model cohort.

With `RUN_CAPRI_RMSD=1`, pyDock prepares `_ref.pdb` and writes the reference
comparison to `<configuration>.RMSD.capri`. This dedicated file is the result
to retain for CAPRI analysis. The `.ene` table can also contain an `RMSD`
(pyDock3) or `l_RMSD` (compatible pyDock4) column calculated during `bindEy`.
That ligand RMSD can differ slightly from the value recalculated by
`capriRMSD`, and the energy table does not replace the CAPRI file because it
lacks `i_RMSD`, FNC and the quality class.

The historical pyDock3 output contains:

```text
Conf  l_RMSD  i_RMSD  FNC  CAPRI
```

Its historical CAPRI class is encoded as an empty value for Incorrect, `*`
for Acceptable, `**` for Medium and `***` for High. Compatible pyDock4 releases
produce this default format instead:

```text
Conf  l_RMSD  i_RMSD  FNC  DockQ  CAPRI
```

Here FNC is the native-contact fraction from 0 to 1, `DockQ` combines FNC,
`i_RMSD` and `l_RMSD`, and the final field contains the DockQ-based textual
class `Incorrect`, `Acceptable`, `Medium` or `High`. The formats and class
semantics therefore depend on the installed pyDock version. Section 3.4 reads
both formats, normalizes their quality labels and adds the metrics to its model
tables without changing the confidence–energy ranking.

## Parallel execution with Greasy

All pyDock3 commands are stored in:

```
4_Case_Studies/${CASE}/Greasy_BindE_mul_ligs.txt
```

The script launches Greasy after generating the file. The saved task list can
also be run again manually from the repository root:

```bash
$GREASY/greasy 4_Case_Studies/${CASE}/Greasy_BindE_mul_ligs.txt
```

Every generated `bindEy` invocation is limited to five minutes:

```text
timeout 5m <PYDOCK>/<PYDOCK_BINARY> <configuration-name> bindEy
```

When reference assessment is enabled, `capriRMSD` is chained to a successful
`bindEy` calculation in the same Greasy task and receives its own five-minute
limit:

```text
timeout 5m <PYDOCK>/<PYDOCK_BINARY> <configuration-name> bindEy && timeout 5m <PYDOCK>/<PYDOCK_BINARY> <configuration-name> capriRMSD
```

This ordering is required because `capriRMSD` uses files prepared by `bindEy`.
An unusually long `bindEy` calculation can indicate severe steric clashes or
another problematic model or configuration. Without a time limit, a stalled
calculation can keep a Greasy worker occupied and delay—or, if workers become
blocked, stop—the remaining queue. This practical limit terminates that task
after five minutes so Greasy can continue scheduling other commands. A
timed-out task exits unsuccessfully and may not produce a complete `.ene` file;
inspect the Greasy log, the input model and its chain configuration before
retrying it.

## Troubleshooting

| Issue                   | Likely Cause                            |
| ----------------------- | --------------------------------------- |
| No `INI` files created | Wrong chain definitions or empty folder |
| Greasy does not start   | `GREASY_HOME` is incorrect, or `GREASY` does not point to the directory containing `greasy` |
| pyDock3 errors          | Missing residue names or invalid PDB    |
| SCWRL3 cannot be started | Incorrect `SCWRL` path in `$PYDOCK/etc/pydock.conf` or incompatible binary architecture |
| No PDB files found      | Incorrect AF2/AF3 folder structure or filename pattern |
| AlphaFold Server conversion fails | Missing Python/Biopython, an unavailable `cif_to_pdb.py`, or a CIF that cannot be represented safely as PDB |
| Duplicate receptor/ligand configuration | Two definitions normalize to the same `_LIG_*_REC_*` output suffix |
| Reference configuration is rejected | Set `RUN_CAPRI_RMSD=1`, provide an existing nonempty `REFERENCE_PDB`, and define one correctly paired reference-chain entry per model-chain entry |
| No `.RMSD.capri` file | `bindEy` or `capriRMSD` failed or timed out; inspect the Greasy log, chain mapping and auxiliary files. Section 3.4 can still retain the `.ene` RMSD as `l_RMSD`, with the remaining reference fields empty |
| A task exits after five minutes | The practical `timeout 5m` limit was reached; inspect the model for clashes and review the Greasy log before retrying |
| Section 3.4 reports multiple energies for one model | More than one receptor–ligand `.ene` configuration maps to the same primary model |
