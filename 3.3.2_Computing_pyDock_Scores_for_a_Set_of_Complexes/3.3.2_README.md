# 📘 pyDock3 Multi-Ligand Batch Pipeline

[![Shell Script](https://img.shields.io/badge/Script-Bash-4EAA25?logo=gnu-bash\&logoColor=white)]()
[![pyDock3](https://img.shields.io/badge/pyDock3-Docking%20Engine-1f77b4)]()
[![AlphaFold2](https://img.shields.io/badge/AlphaFold2-Models-blue)]()
[![Greasy Scheduler](https://img.shields.io/badge/Greasy-Job%20Scheduler-9c27b0)]()

Automated workflow for generating **pyDock3 configuration files** (`INI`) from
AlphaFold2 and AlphaFold3 prediction folders and launching large-scale
binding-energy evaluations using **Greasy**. The script builds one or multiple
receptor–ligand configurations per model and schedules all pyDock3 runs in
parallel. If an input model has incomplete side chains, pyDock3 can use the
SCWRL3 executable configured in its own `etc/pydock.conf`; the shell script
does not invoke SCWRL directly.

---

## 📌 Features

* Automatic discovery of AF2 ColabFold, AF3 Server and local AF3 prediction folders
* Detection of AF2 `*[0-9].pdb` and AF3 `[0-9]*_model.pdb` models
* Multi-ligand, multi-configuration `INI` generation
* Fully automated pyDock3 command list creation
* Greasy-powered parallel energy calculations
* Environment-variable customization of receptors, ligands, paths, and workers

---

## 📁 Workflow Summary

```
AF2/AF3 Models → Chain Definitions → pyDock3 INI Files → Greasy Job List → Binding-Energy Calculations
```

`ini_creator_bindE.sh` processes one case per invocation. It derives the
repository root from the script location, independently of the caller's current
working directory. For the selected case it scans:

* `Case_Studies_Included/<CASE>/AF2/*cola*v*/` for AF2 `*[0-9].pdb` files;
* `Case_Studies_Included/<CASE>/AF3/fold*/` for AlphaFold Server models;
* `Case_Studies_Included/<CASE>/AF3/output/<CASE>/seed-*/` for local AF3
  `[0-9]*_model.pdb` files.

# 🔧 Input Variables

The script uses several environment variables, each with a default value.
Override them as needed before execution.

---

## **1. `CASE` — Case Identifier**

**Purpose:**
Selects the case to search
(`<repo_root>/Case_Studies_Included/${CASE}/`).

**Default:**

```bash
4POU
```

**Override:**

```bash
export CASE="MyProject"
```

---

## **2. `PYDOCK` — Path to pyDock3**

**Purpose:**
Specifies the directory containing the `pyDock3` executable. The script calls
`${PYDOCK}/pyDock3` directly.

**Default:**

```
/usr/local/software/pyDock3/
```

**Override:**

```bash
export PYDOCK="/opt/pydock3"
```

Installation, license, compatibility and verification instructions are in
[section 2.3](../2.3_pyDock_Scoring/2.3_README.md).

### SCWRL3 used internally by pyDock3

`ini_creator_bindE.sh` does not require a separate `SCWRL` environment
variable and does not call SCWRL itself. When side-chain reconstruction is
needed, pyDock3 reads the executable path from its configuration file:

```ini
# $PYDOCK/etc/pydock.conf
[apps]
SCWRL=/absolute/path/to/pyDock3/3.0/scwrl3
```

The pyDock3 distribution used by this workflow includes the SCWRL3 archive.
Follow [section 2.4](../2.4_SCWRL/2.4_README.md) to install that bundled copy
and configure its absolute path. SCWRL4 should not be substituted in this
configuration unless the installed pyDock version explicitly supports it.

---

## **3. `GREASY` — Directory Containing the Greasy Wrapper**

**Purpose:**
Defines the directory containing the public `greasy` wrapper. A standard
installation places it in `<installation_prefix>/bin/greasy`.

**Default:**

```
/path/to/software/greasy/
```

**Override:**

```bash
export GREASY_HOME="/opt/greasy/GREASY_2.2"
export GREASY="${GREASY_HOME}/bin"
```

This convention matches the installation documented in
[section 2.5](../2.5_Greasy/2.5_README.md). `ini_creator_bindE.sh` invokes
`${GREASY}/greasy` directly, so `GREASY` must name the directory containing the
wrapper rather than only the installation prefix.

---

## **4. `GREASY_NWORKERS` — Number of Parallel Workers**

**Default:**

```bash
8
```

Set based on CPU/GPU availability:

```bash
export GREASY_NWORKERS=4
```

---

## **5. `CHAINS_REC_LIG_VALUES` — Receptor/Ligand Chain Definitions**

**Purpose:**
Controls how chains inside each PDB model are assigned to **receptor** and **ligand** when generating `INI` files for pyDock3.

Each entry in this array must follow:

```
"RECEPTOR_CHAINS LIGAND_CHAINS"
```

### 📌 Interpretation Rules

* Text **before** the space → receptor chain(s)
* Text **after** the space → ligand chain(s)
* Multiple chains are allowed via commas (`A,B`)
* **Each array entry produces one `INI` file per PDB model**

➡️ If you define **N entries**, the script will generate **N `INI` files per model**.

---

# 🧬 Chain Definition Examples

### **1️⃣ Simple Case**

```bash
CHAINS_REC_LIG_VALUES=("A B" "A C")
source ./ini_creator_bindE.sh
```

Generates **two** `INI` files for every PDB model:

* Receptor = A, Ligand = B → `*_LIG_B.ini`
* Receptor = A, Ligand = C → `*_LIG_C.ini`

Multiple values are a Bash array and are therefore used by sourcing the script
from this section directory; Bash arrays are not inherited by a child process
started with `bash ini_creator_bindE.sh`. A single definition can be exported
normally, as shown below.

---

### **2️⃣ Multiple Receptor Chains**

```bash
export CHAINS_REC_LIG_VALUES="A,B C"
```

Interpretation:

* Receptor = A,B
* Ligand = C

Generates **one** `INI` file per PDB:

```
*_LIG_C.ini
```

---

### 🧾 Summary Table

| Definition             | Receptor | Ligand | Result                        |
| ---------------------- | -------- | ------ | ----------------------------- |
| `"A B"`                | A        | B      | 1 `INI`                      |
| `"A C"`                | A        | C      | 1 `INI`                      |
| `"A,B C"`              | A,B      | C      | 1 `INI`                      |
| Multiple array entries | —        | —      | Multiple `INI` files per PDB |

# 📂 Output Structure

After execution, the script produces:

```
Case_Studies_Included/${CASE}/
├── AF2/
│   ├── *colab*v*/
│   │   ├── *[0-9].pdb
│   │   └── generated *_LIG_*.ini
│   └── Greasy_BindE_mul_ligs.txt     # Combined Greasy command list
└── AF3/
    ├── fold*/                        # AlphaFold Server output
    │   ├── [0-9]*_model.pdb
    │   └── generated *_LIG_*.ini
    └── output/${CASE}/seed-*/        # Local AlphaFold 3 output
        ├── [0-9]*_model.pdb
        └── generated *_LIG_*.ini
```

---

# ⚙️ Parallel Execution with Greasy

All pyDock3 commands are stored in:

```
Case_Studies_Included/${CASE}/AF2/Greasy_BindE_mul_ligs.txt
```

The script launches Greasy automatically after generating the file. Each task
uses an absolute prediction-directory path, so neither generation nor execution
depends on the original working directory. From the repository root, the saved
task file can also be run again manually:

```bash
$GREASY/greasy Case_Studies_Included/${CASE}/AF2/Greasy_BindE_mul_ligs.txt
```

# ❗ Troubleshooting

| Issue                   | Likely Cause                            |
| ----------------------- | --------------------------------------- |
| No `INI` files created | Wrong chain definitions or empty folder |
| Greasy does not start   | `GREASY` does not point to the directory containing `greasy` |
| pyDock3 errors          | Missing residue names or invalid PDB    |
| SCWRL3 cannot be started | Incorrect `SCWRL` path in `$PYDOCK/etc/pydock.conf` or incompatible binary architecture |
| No PDB files found      | Incorrect AF2/AF3 folder structure or filename pattern |
