# Combined Model Confidence and pyDock Score

This directory contains the notebook used in section 3.4:

- **Analysis_protocol.ipynb** consolidates AlphaFold2/ColabFold and local
  AlphaFold3 predictions with precomputed pyDock energies.
- The notebook calculates energy rankings, confidence Z-scores and a combined
  confidence–energy ranking for one case study at a time.

The workflow does not require an experimental reference and does not calculate
RMSD. It also does not run AlphaFold, structural relaxation or pyDock; those
steps must already have produced the inputs described below.

## Workflow summary

~~~text
AF2 models + AF2 log.txt + pyDock.ene
                              \
                               -> validated model table
                              /
AF3 models + summary JSON + pyDock.ene
                               -> per-case Z-scores
                               -> combined confidence/energy ranking
                               -> CSV tables and plots
~~~

The native `Total` field from each pyDock energy file is exposed in the output
tables as **pyDock-0.1VDW**, following the chapter nomenclature. The source
`.ene` files are read-only and are not renamed or modified.

## Requirements

The notebook requires a Python environment with:

- Python 3
- Jupyter
- pandas
- NumPy
- Matplotlib

It expects AlphaFold and pyDock results under the repository's
Case_Studies_Included directory.

## Expected directory layout

For a case identified by CASE_ID, the relevant layout is:

~~~text
Case_Studies_Included/<CASE>/
├── AF2/
│   ├── <CASE>.colabfold.v1/
│   │   ├── log.txt
│   │   ├── *_unrelaxed_*.pdb
│   │   ├── *_LIG_*.ini
│   │   └── *_LIG_*.ene
│   ├── <CASE>.colabfold.v2/
│   └── <CASE>.colabfold.v3/
├── AF3/
│   └── output/<CASE>/
│       ├── <CASE>_summary_confidences.json
│       └── seed-*_sample-*/
│           ├── *_model.pdb
│           ├── *_summary_confidences.json
│           ├── *_model_LIG_*.ini
│           └── *_model_LIG_*.ene
└── Analysis/
    └── notebook outputs
~~~

The AF3 summary at the root of output/<CASE>/ is an aggregate best-model
summary in the current local output and is not treated as an additional model.
Only summary JSON files inside seed/sample directories are loaded.

## Configuration and execution

The main configuration is at the beginning of Analysis_protocol.ipynb.

| Setting | Default | Purpose |
| --- | --- | --- |
| CASE_ID | 4POU | Case directory to analyse. |
| INCLUDE_AF2_UNRELAXED | True | Include AF2 models labelled unrelaxed. |
| INCLUDE_AF2_RELAXED | False | Include relaxed AF2 models when available. |
| INCLUDE_AF3_AS_PREDICTED | True | Include local AF3 model PDB files as predicted. |
| DROP_DUPLICATE_AF2_FINALS | True | Remove unsuffixed AF2 finals that duplicate recycle 20. |
| GENERATE_PLOTS | True | Write the three lightweight PNG plots. |
| REPO_ROOT_OVERRIDE | None | Optional explicit repository root. |

The notebook searches upwards from the current working directory for the
repository. If it is launched from elsewhere, set REPO_ROOT_OVERRIDE or the
AF_PYDOCK_REPO_ROOT environment variable.

From the repository root, it can be opened with a command such as:

~~~bash
jupyter notebook \
  3.4_Combined_Model_Confidence_and_pyDock_Score/Analysis_protocol.ipynb
~~~

The output directory is created automatically:

~~~text
Case_Studies_Included/<CASE>/Analysis/
~~~

## Extraction and matching protocol

### 1. pyDock energy files

The notebook discovers:

- AF2 energy files recursively below Case_Studies_Included/<CASE>/AF2/.
- Local AF3 energy files in
  Case_Studies_Included/<CASE>/AF3/output/<CASE>/seed-*_sample-*/.

Each .ene table is expected to contain:

~~~text
Conf  Ele  Desolv  VDW  Total  RANK
~~~

The notebook reads Ele, Desolv, VDW and Total. Conf and the per-file RANK are
not exported because each file in the current workflow contains one scored
model and those fields are therefore not global model rankings.

The output energy columns use the chapter names:

~~~text
pyDock-0.1VDW = native .ene Total
               = Ele + Desolv + 0.1 × VDW

pyDock-1VDW   = Ele + Desolv + VDW
~~~

Small differences between the displayed formula and the .ene Total value can
result from decimal rounding in the energy file.

### 2. Associating an energy file with its model PDB

Replacing the .ene suffix with .pdb is not sufficient because energy files
normally end in _LIG_<CHAIN>.ene.

The notebook therefore:

1. Reads the companion .ini file and uses its pdb entry when available.
2. Otherwise removes the explicit _LIG_<CHAIN> suffix.
3. Verifies that the resulting PDB exists.
4. Rejects pyDock auxiliary files ending in _rec.pdb or _lig.pdb.
5. Records the successful method in pdb_resolution.

An energy record without a verified primary PDB is reported and excluded from
the ranking.

### 3. AlphaFold2 confidence from log.txt

AF2 logs are discovered with:

~~~text
Case_Studies_Included/<CASE>/AF2/*/log.txt
~~~

The reader extracts recycle records resembling:

~~~text
alphafold2_multimer_v1_model_1_seed_000 recycle=20 \
pLDDT=91.4 pTM=0.767 ipTM=0.590 tol=0.182
~~~

For every matching line it extracts:

- AF2-Multimer version
- model number
- seed
- recycle number
- pLDDT
- pTM
- ipTM
- tol, when present

AF2 energy and confidence records are matched explicitly by case, prediction
method, AF version, model number, seed and recycle. The structural state is
taken from the PDB filename; it is never forced to relaxed.

### 4. Local AlphaFold3 confidence from JSON

AF3 confidence files are discovered only with:

~~~text
Case_Studies_Included/<CASE>/AF3/output/<CASE>/
  seed-*_sample-*/*_summary_confidences.json
~~~

The reader extracts available values for:

- pTM (from the native JSON key `ptm`)
- ipTM (from the native JSON key `iptm`)
- `ranking_score`
- `fraction_disordered`
- `has_clash`
- `num_recycles`
- chain identifiers and chain-level confidence arrays

Local AF3 summary files do not necessarily provide pLDDT or num_recycles.
Missing method-specific values remain empty rather than being invented.

AF3 energy and confidence records are matched by case, prediction method,
AF version, `model_id`, seed and sample.

### 5. Model selection and duplicate handling

The AF2 and AF3 filename parsers are independent:

- AF2 names encode state, AF version, AF rank, model number, seed and optional
  recycle.
- Local AF3 names encode case, seed and sample and end in _model.pdb.

The default cohort:

- includes AF2 recycled unrelaxed models;
- excludes AF2 relaxed models unless explicitly enabled;
- includes local AF3 _model.pdb files with relaxation_status set to
  as_predicted;
- excludes pyDock receptor and ligand helper PDB files.

An unsuffixed AF2 final model is removed only when both its PDB and energy file
are byte-identical to the corresponding .r20 files. This avoids counting the
same prediction twice without discarding a genuinely different final model.

A model enters the consolidated matched set only when it has:

- a verified primary PDB;
- a pyDock .ene file;
- matching AF2 or AF3 confidence data.

## Scores, Z-scores and rankings

AF2 and AF3 records are merged first. Z-scores and energy rankings are then
calculated across all selected models of the same case, not separately by
prediction method or AF version.

For a metric x:

~~~text
z = (x - case mean) / case population standard deviation
~~~

The implementation uses the population standard deviation with ddof=0.
Missing values remain missing. If a valid cohort has zero variance, its
Z-score is set to zero.

AlphaFold model confidence uses the chapter abbreviation:

~~~text
AF-MC = 0.8 × ipTM + 0.2 × pTM
~~~

Energy rankings use ascending order because lower energy is favourable:

~~~text
rank_pyDock-0.1VDW = rank of pyDock-0.1VDW within the case
rank_pyDock-1VDW   = rank of pyDock-1VDW within the case
~~~

The primary combined score follows the chapter equation:

~~~text
Z_AF/pyDock-1VDW = Z_AF-MC - Z_pyDock-1VDW
~~~

Higher confidence increases this score, while a favourable low
pyDock-1VDW energy also increases it through subtraction of its Z-score.
`rank_Z_AF/pyDock-1VDW` orders `Z_AF/pyDock-1VDW` from highest to lowest.

The chapter defines the metric names but not literal CSV headers for their
ranks. This repository therefore uses the explicit convention
`rank_<metric>` for derived ranking columns.

## Output files

Each execution overwrites the current outputs for the configured case:

| File | Contents |
| --- | --- |
| consolidated_results.csv | All models with verified PDB, energy and matching confidence data, including calculated metrics and rankings. |
| combined_ranking.csv | Rankable models ordered by `rank_Z_AF/pyDock-1VDW`. |
| metric_distributions.png | Histograms of AF-MC, pyDock-0.1VDW, pyDock-1VDW and Z_AF/pyDock-1VDW, separated visually by AF2/AF3. |
| metric_density.png | Density-style histograms of the confidence, energy and combined normalized metrics. |
| confidence_vs_pydock.png | AF-MC plotted against pyDock-1VDW. |

Paths stored in the CSV files are relative to the repository root whenever
possible, making the tables portable with the repository.

## Column reference

### Identification and paths

| Column | Meaning |
| --- | --- |
| case_id | Case-study identifier, for example 4POU. |
| model_id | Stable model identifier derived from the verified primary PDB filename. |
| prediction_method | AF2 or AF3. |
| af_version | AF2-Multimer version or alphafold3. |
| relaxation_status | unrelaxed, relaxed or as_predicted. |
| model_path | Repository-relative path to the primary PDB. |
| energy_path | Repository-relative path to the source .ene file. |
| confidence_path | Path to the AF2 log.txt or AF3 summary confidence JSON. |

### pyDock energies and rankings

| Column | Meaning |
| --- | --- |
| Ele | Electrostatic energy read from the .ene file. |
| Desolv | Desolvation energy read from the .ene file. |
| VDW | Full van der Waals term read from the .ene file. |
| pyDock-0.1VDW | Native `.ene` `Total` score, using 0.1 × VDW. |
| pyDock-1VDW | Recalculated energy using the full VDW contribution. |
| rank_pyDock-0.1VDW | Global ascending rank of pyDock-0.1VDW within the case. |
| rank_pyDock-1VDW | Global ascending rank of pyDock-1VDW within the case. |

### Confidence and combined ranking

| Column | Meaning |
| --- | --- |
| pLDDT | AF2 per-model pLDDT extracted from `log.txt`; normally absent for local AF3 summary JSON. |
| pTM | Predicted TM score from the AF2 log or AF3 summary JSON. |
| ipTM | Interface predicted TM score from the AF2 log or AF3 summary JSON. |
| ranking_score | Native AF3 ranking score; normally absent for AF2 recycle records. |
| AF-MC | 0.8 × ipTM + 0.2 × pTM. |
| Z_AF-MC | Per-case Z-score of AF-MC. |
| Z_pyDock-0.1VDW | Per-case Z-score of pyDock-0.1VDW. |
| Z_pyDock-1VDW | Per-case Z-score of pyDock-1VDW. |
| Z_AF/pyDock-1VDW | Z_AF-MC minus Z_pyDock-1VDW. |
| rank_Z_AF/pyDock-1VDW | Descending rank of Z_AF/pyDock-1VDW. |

### AF2 and AF3 model metadata

| Column | Meaning |
| --- | --- |
| af2_filename_rank | AF2/ColabFold final-model position encoded in `_rank_NNN_` in the PDB filename; this is not a pyDock or combined rank. |
| model_number | AF2 model number. |
| seed | AF2 or AF3 random seed parsed from the input record or filename. |
| recycle | AF2 recycling iteration. |
| sample | Local AF3 sample number. |
| tol | AF2 recycle tolerance when present in log.txt. |

Integer metadata and ranking fields are written without decimal suffixes.
Fields that do not apply to a prediction method remain empty; for example,
`af2_filename_rank`, `model_number` and `recycle` are empty for AF3 rather than being
assigned artificial values.

### Additional AF3 confidence fields

| Column | Meaning |
| --- | --- |
| fraction_disordered | AF3 predicted disordered fraction when present. |
| has_clash | AF3 clash indicator when present. |
| num_recycles | Optional AF3 recycle count; it can be absent from summary JSON. |
| chain_ids | Ordered unique AF3 chain identifiers, serialized as JSON in the CSV. |
| chain_ptm | AF3 per-chain pTM array, serialized as JSON. |
| chain_pair_iptm | AF3 chain-pair ipTM matrix, serialized as JSON. |
| chain_A_pTM | First chain pTM value, corresponding to chain A for the current 4POU layout. |
| chain_B_pTM | Second chain pTM value, corresponding to chain B for the current 4POU layout. |
| ipTM_A_vs_B | First off-diagonal chain-pair ipTM value for the current two-chain layout. |

### Diagnostics

| Column | Meaning |
| --- | --- |
| pdb_resolution | How the model PDB was found: ini or validated_suffix. |

## Limitations and practical notes

- Only local AF3 seed/sample output is currently read. AlphaFold Server output
  must first be arranged in a compatible model/confidence layout or supported
  by an additional reader.
- No experimental structure, RMSD or CAPRI metrics are required.
- The notebook does not generate models, relax structures or execute pyDock.
- Method-specific empty values are expected. For example, pLDDT and tol are
  AF2-derived, while ranking_score and chain-level arrays are AF3-derived.
- The derived `chain_A_pTM`, `chain_B_pTM` and `ipTM_A_vs_B` fields assume the first
  two entries represent chains A and B. The original arrays are retained for
  cases with different or more complex chain organization.
- Re-running the notebook overwrites the five files in the case Analysis
  directory.
