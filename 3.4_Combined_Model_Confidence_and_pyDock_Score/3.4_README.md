# Combined Model Confidence and pyDock Score

This directory contains the notebook used in section 3.4:

- **Analysis_protocol.ipynb** consolidates AlphaFold2/ColabFold and AlphaFold3
  local or Server predictions with precomputed pyDock energies.
- The notebook calculates energy rankings, confidence Z-scores and a combined
  confidence–energy ranking for one case study at a time.

The workflow does not require an experimental reference and does not calculate
RMSD. It also does not run AlphaFold, structural relaxation or pyDock; those
steps must already have produced the inputs described below.

In particular, complete the [section 3.3.2 pyDock scoring
workflow](../3.3.2_Computing_pyDock_Scores_for_a_Set_of_Complexes/3.3.2_README.md)
before running this notebook. Section 3.4 reads the resulting `.ene` files; it
does not calculate missing pyDock energies.

## Workflow summary

~~~text
AF2 models + AF2 log.txt + pyDock.ene
                              \
                               -> validated model table
                              /
AF3 local/Server models + summary JSON + pyDock.ene
                               -> per-case Z-scores
                               -> combined confidence/energy ranking
                               -> CSV tables and plots
~~~

The native `Total` field from each pyDock energy file is exposed in the output
tables as **pyDock-0.1VDW**, following the chapter nomenclature. The source
`.ene` files are read-only and are not renamed or modified.

## Python environment

The notebook imports NumPy, pandas, Matplotlib, ipywidgets and py3Dmol; it does
not import or run AlphaFold, ColabFold or pyDock. Those programs only need to
have produced the input files under `4_Case_Studies/`.

The root [`environment.yml`](../environment.yml) defines Python 3.12 and the
required analysis packages. It also includes `ipykernel`, the component needed
to use the environment as a notebook kernel, ipywidgets for the Top-N controls,
Jinja2 for the styled summary table, py3Dmol for molecular visualization and
Notebook 7 for launching the document from a terminal. IPython is installed
transitively by `ipykernel` and does not need to be listed separately.

### Independent environment (recommended)

From the repository root:

```bash
conda env create --file environment.yml
conda activate af-pydock-analysis
```

The name in the YAML can be edited, or overridden on the command line:

```bash
conda env create --name my-analysis-env --file environment.yml
conda activate my-analysis-env
```

### Reusing an existing environment

The same dependencies can be added to an existing environment when avoiding a
separate environment is more important than isolation. First review the
`python=` constraint in `environment.yml` and adjust it if the existing
software requires another compatible Python version. For example:

```bash
conda env update --name Alphafold3 --file environment.yml
conda activate Alphafold3
```

The command-line name overrides `name: af-pydock-analysis` from the YAML. Do
not use `--prune`, because AlphaFold3 packages are intentionally absent from
this lightweight file. Review the proposed dependency changes before updating
an important working installation; Conda may otherwise replace packages used
by the local AF3 runtime.

In VS Code, select the chosen Conda environment as the notebook kernel. The
repository does not require an AF2 environment merely to read and analyse
existing supported result files.

## Expected directory layout

For a case identified by CASE_ID, the relevant layout is:

~~~text
4_Case_Studies/<CASE>/
├── AF2/
│   ├── <CASE>.colabfold.v1/
│   │   ├── log.txt
│   │   ├── *_relaxed_*.pdb          # recommended AF2 input
│   │   ├── *_unrelaxed_*.pdb        # technical testing only
│   │   ├── *_LIG_*_REC_*.ini
│   │   └── *_LIG_*_REC_*.ene
│   ├── <CASE>.colabfold.v2/
│   └── <CASE>.colabfold.v3/
├── AF3/
│   ├── output/<CASE>/
│   │   ├── <CASE>_summary_confidences.json
│   │   └── seed-*_sample-*/
│   │       ├── *_model.pdb
│   │       ├── *_summary_confidences.json
│   │       ├── *_model_LIG_*_REC_*.ini
│   │       └── *_model_LIG_*_REC_*.ene
│   └── fold*/
│       ├── *_model_<N>.pdb
│       ├── *_summary_confidences_<N>.json
│       ├── *_model_<N>_LIG_*_REC_*.ini
│       └── *_model_<N>_LIG_*_REC_*.ene
└── Analysis/
    └── notebook outputs
~~~

The AF3 summary at the root of `output/<CASE>/` is an aggregate best-model
summary in the current local output and is not treated as an additional model.
Local summaries are loaded only from seed/sample directories. Indexed AF3
Server summaries are loaded from each `fold*/` directory.

Relaxed AF2 structures are recommended for the pyDock scoring and combined
analysis workflow. `run_alphafold2_multimer.sh` relaxes the final ranked models
during generation, whereas its saved recycle snapshots remain unrelaxed. Use
`relax_alphafold2_models.sh` to prepare the complete relaxed recycle ensemble,
then run pyDock section 3.3.2 for the relaxed PDB files. Unrelaxed AF2 models
may be enabled temporarily to test the notebook readers and calculations, but
that test is not the recommended protocol analysis.

## Configuration and execution

The main configuration is at the beginning of Analysis_protocol.ipynb.

| Setting | Default | Purpose |
| --- | --- | --- |
| CASE_ID | 4POU | Case directory to analyse. |
| INCLUDE_AF2_UNRELAXED | False | Exclude unrelaxed AF2 models from the recommended analysis. Enable only for a technical test if needed. |
| INCLUDE_AF2_RELAXED | True | Include the recommended relaxed AF2 models after they have been relaxed and scored with pyDock. |
| INCLUDE_AF3_AS_PREDICTED | True | Include available local and Server AF3 model PDB files as predicted. |
| DROP_DUPLICATE_AF2_FINALS | True | Remove unsuffixed AF2 finals that duplicate recycle 20. |
| GENERATE_PLOTS | True | Write the three lightweight PNG plots. |
| DISPLAY_PLOTS | True | Display the three Matplotlib figures directly in the notebook. |
| DEFAULT_TOP_N | 10 | Initial number of models in the static and interactive Top-N tables. |
| MAX_TOP_N | 50 | Maximum value offered by the interactive Top-N slider. |
| REPO_ROOT_OVERRIDE | None | Optional explicit repository root. |

The notebook searches upwards from the current working directory for the
repository. If it is launched from elsewhere, set REPO_ROOT_OVERRIDE or the
AF_PYDOCK_REPO_ROOT environment variable.

`CASE_ID` remains an explicit manual setting. Run the notebook separately for
each case that needs its own normalized scores and output files.

The defaults enforce the recommended relaxed-AF2 workflow. Before running the
complete analysis, use the relaxed final models generated by
`run_alphafold2_multimer.sh` and, for the complete recycle ensemble, prepare the
saved recycles with `relax_alphafold2_models.sh`. Generate the corresponding
`.ene` files with section 3.3.2. To test the notebook before those steps are
complete, temporarily set `INCLUDE_AF2_UNRELAXED=True` and
`INCLUDE_AF2_RELAXED=False`; results from that configuration are a software
test, not the final protocol output. With the default settings, the notebook
stops explicitly if it cannot find relaxed AF2 models with associated pyDock
`.ene` files.

For an analysis containing only AF3 models, set both
`INCLUDE_AF2_UNRELAXED=False` and `INCLUDE_AF2_RELAXED=False`. This disables
the AF2 cohort and therefore requires neither AF2 outputs nor an AF2 software
installation. Keep `INCLUDE_AF3_AS_PREDICTED=True` and provide local and/or
Server AF3 confidence JSON and pyDock `.ene` files as described below.

From the repository root, it can be opened with a command such as:

~~~bash
jupyter notebook \
  3.4_Combined_Model_Confidence_and_pyDock_Score/Analysis_protocol.ipynb
~~~

The output directory is created automatically:

~~~text
4_Case_Studies/<CASE>/Analysis/
~~~

## Extraction and matching protocol

### 1. pyDock energy files

The notebook discovers:

- AF2 energy files recursively below 4_Case_Studies/<CASE>/AF2/.
- Local AF3 energy files in
  4_Case_Studies/<CASE>/AF3/output/<CASE>/seed-*_sample-*/.
- AF3 Server energy files directly below
  4_Case_Studies/<CASE>/AF3/fold*/.

Each .ene table is expected to contain:

~~~text
Conf  Ele  Desolv  VDW  Total  RANK
~~~

The notebook reads Ele, Desolv, VDW and Total. Conf and the per-file RANK are
not exported because each file in the current workflow contains one scored
model and those fields are therefore not global model rankings.

The output energy columns use the chapter names:

~~~text
pyDock-0.1VDW = native .ene Total
               = Ele + Desolv + 0.1 × VDW

pyDock-1VDW   = Ele + Desolv + VDW
~~~

Small differences between the displayed formula and the .ene Total value can
result from decimal rounding in the energy file.

### 2. Associating an energy file with its model PDB

The notebook associates each `.ene` file with its primary model PDB, using the
companion `.ini` file when available. Both the current
`_LIG_<LIGAND>_REC_<RECEPTOR>.ene` names and the shorter
`_LIG_<LIGAND>.ene` names are supported. The selected PDB must exist, and
pyDock auxiliaries ending in `_rec.pdb` or `_lig.pdb` are not accepted as
prediction models. The `pdb_resolution` column records how the association was
resolved.

An energy record without a verified primary PDB is reported and excluded from
the ranking. Exactly one `.ene` file may map to each primary PDB in the current
workflow. If several receptor–ligand configurations produce multiple `.ene`
files for the same model, keep only the configuration intended for this
analysis; otherwise the notebook reports the ambiguity and stops.

### 3. AlphaFold2 confidence from log.txt

AF2 logs are discovered with:

~~~text
4_Case_Studies/<CASE>/AF2/*/log.txt
~~~

The reader extracts recycle records resembling:

~~~text
alphafold2_multimer_v1_model_1_seed_000 recycle=20 \
pLDDT=91.4 pTM=0.767 ipTM=0.590 tol=0.182
~~~

For every matching line it extracts:

- AF2-Multimer version
- model number
- seed
- recycle number
- pLDDT
- pTM
- ipTM
- tol, when present

AF2 energy and confidence records are matched explicitly by case, prediction
method, AF version, model number, seed and recycle. The structural state is
taken from the PDB filename; it is never forced to relaxed.

### 4. AlphaFold3 confidence from JSON

Local AF3 confidence files are discovered with:

~~~text
4_Case_Studies/<CASE>/AF3/output/<CASE>/
  seed-*_sample-*/*_summary_confidences.json
~~~

AF3 Server confidence files are discovered with:

~~~text
4_Case_Studies/<CASE>/AF3/fold*/
  *_summary_confidences_<N>.json
~~~

The reader extracts available values for:

- pTM (from the native JSON key `ptm`)
- ipTM (from the native JSON key `iptm`)
- `ranking_score`
- `fraction_disordered`
- `has_clash`
- `num_recycles`
- chain identifiers and chain-level confidence arrays

AF3 summary files do not necessarily provide pLDDT, num_recycles or chain
identifiers. Missing method-specific values remain empty, while any available
chain-level confidence arrays are retained.

AF3 energy and confidence records are matched by case, prediction method,
AF version and `model_id`. Local seed and sample values remain available as
metadata but are not invented for Server models.

### 5. Model selection and duplicate handling

The AF2 and AF3 filename parsers are independent:

- AF2 names encode state, AF version, AF rank, model number, seed and optional
  recycle.
- Local AF3 names encode case, seed and sample and end in _model.pdb.
- AF3 Server names encode the fold prefix and model index and end in
  `_model_<N>.pdb`.

The recommended cohort:

- includes AF2 recycled relaxed models;
- excludes unrelaxed AF2 models;
- includes local and Server AF3 model PDB files with `relaxation_status` set
  to `as_predicted`;
- excludes pyDock receptor and ligand helper PDB files.

The AF2 generation command requests Amber relaxation for the final ranked
models. It also saves unrelaxed recycle snapshots; the unsuffixed unrelaxed
final represents the same pre-relaxation prediction as its `.r20` file. With
`DROP_DUPLICATE_AF2_FINALS=True`, an equivalent unrelaxed final and `.r20`
record are counted once. The expanded recycle ensemble is relaxed afterward
with `relax_alphafold2_models.sh`.

A model enters the consolidated matched set only when it has:

- a verified primary PDB;
- a pyDock .ene file;
- matching AF2 or AF3 confidence data.

## Scores, Z-scores and rankings

AF2 and AF3 records are merged first. Z-scores and energy rankings are then
calculated across all selected models of the same case, not separately by
prediction method or AF version.

For a metric x:

~~~text
z = (x - case mean) / case population standard deviation
~~~

Z-scores use the population standard deviation (`ddof=0`) for the selected
case cohort. Missing values remain missing. If a valid cohort has zero
variance, its Z-score is set to zero.

AlphaFold model confidence uses the chapter abbreviation:

~~~text
AF-MC = 0.8 × ipTM + 0.2 × pTM
~~~

Energy rankings use ascending order because lower energy is favourable:

~~~text
rank_pyDock-0.1VDW = rank of pyDock-0.1VDW within the case
rank_pyDock-1VDW   = rank of pyDock-1VDW within the case
~~~

The primary combined score follows the chapter equation:

~~~text
Z_AF/pyDock-1VDW = Z_AF-MC - Z_pyDock-1VDW
~~~

Higher confidence increases this score, while a favourable low
pyDock-1VDW energy also increases it through subtraction of its Z-score.
`rank_Z_AF/pyDock-1VDW` orders `Z_AF/pyDock-1VDW` from highest to lowest.

The chapter defines the metric names but not literal CSV headers for their
ranks. This repository therefore uses the explicit convention
`rank_<metric>` for derived ranking columns.

## Static summary and interactive Top N

The first view is a static table containing the default Top N according to
`Z_AF/pyDock-1VDW`. After the notebook has been executed and saved, this table
also remains visible in a non-interactive rendering. The widget panel then
provides:

- a ranking-criterion selector;
- an `All`, `AF2` or `AF3` method filter;
- a Top-N slider, initially 10 and limited by `MAX_TOP_N`;
- a `Save current Top N` button for a self-contained model selection;
- a model selector connected to a py3Dmol viewer.

The criteria retain their scientific direction:

| Criterion | Preferred direction |
| --- | --- |
| Z_AF/pyDock-1VDW | Higher is better. |
| Z_AF-MC | Higher is better. |
| Z_pyDock-1VDW | Lower is better. |
| Z_pyDock-0.1VDW | Lower is better. |

All controls operate only on the manually configured `CASE_ID`. Changing the
criterion reorders the selected cohort, while the method selector filters it
before reordering; neither control mixes cases or recomputes Z-scores. One PDB
is displayed at a time to avoid creating many simultaneous molecular viewers.
The method selector remains grouped as `AF2` or `AF3`, while AF2 rows in the
displayed tables identify their Multimer version as `AF2 v1`, `AF2 v2` or
`AF2 v3` when that information is available in `af_version`.

The displayed `selection_rank` is generated only for the current widget
criterion and method filter. It is not written to either CSV and must not be
confused with the persistent combined `rank_Z_AF/pyDock-1VDW` column.

The viewer reads the exact repository-relative `model_path` stored in the
ranking. Before loading it, the notebook verifies that the resolved file is an
existing PDB inside the current case directory. A missing or invalid path is
reported in the viewer area without interrupting the rest of the analysis.

### Self-contained Top-N export

`Save current Top N` captures the table exactly as defined by the current
ranking criterion, method filter and Top-N value. Every click creates a new
UTC-timestamped directory without overwriting an earlier selection:

~~~text
4_Case_Studies/<CASE>/Analysis/selections/
└── <CASE>_<METRIC>_<METHOD>_top<N>_<UTC_TIMESTAMP>/
    ├── selected_ranking.csv
    ├── selection_metadata.json
    └── models/
        ├── AF2_v2/
        ├── AF2_v3/
        └── AF3/
~~~

Only method/version directories represented in the selection are created.

The notebook validates every source PDB before creating the export and then
copies it into the corresponding method/version directory. Copied filenames
are prefixed with `selection_rank` to preserve their order and avoid filename
collisions. The original models are not modified.

`selected_ranking.csv` contains the displayed columns plus
`source_model_path` and the selection-relative `saved_model` path. The JSON
records the case, criterion, direction, method filter, requested Top N, actual
model count and relevant notebook input flags. The resulting directory is
self-contained for inspecting or sharing the selected PDBs and ranking table;
it deliberately does not duplicate the source `.ene`, AF2 logs or AF3
confidence JSON required to recalculate the complete analysis.

ipywidgets require an active notebook kernel. py3Dmol normally loads the
3Dmol.js viewer from a public content-delivery network, so the interactive 3D
view can require internet access even though the PDB itself is local. The
static table, CSV files and Matplotlib plots do not depend on that 3D viewer.

## Output files

Each execution overwrites the two CSV tables for the configured case. The PNG
files are written—and therefore overwrite same-named existing plots—only when
`GENERATE_PLOTS=True` and at least one model has all metrics required for
ranking. `DISPLAY_PLOTS` independently controls whether those same figures are
shown in the notebook:

| File | Contents |
| --- | --- |
| consolidated_results.csv | All models with verified PDB, energy and matching confidence data, including calculated metrics and rankings. |
| combined_ranking.csv | Rankable models ordered by `rank_Z_AF/pyDock-1VDW`. |
| metric_distributions.png | Histograms of AF-MC, pyDock-0.1VDW, pyDock-1VDW and Z_AF/pyDock-1VDW, separated visually by AF2/AF3. |
| metric_density.png | Density-style histograms of the confidence, energy and combined normalized metrics. |
| confidence_vs_pydock.png | AF-MC plotted against pyDock-1VDW. |
| selections/<selection>/ | Timestamped Top-N table, metadata and copied PDB models created only when `Save current Top N` is pressed. |

Paths stored in the CSV files are relative to the repository root whenever
possible, making the tables portable with the repository.

## Column reference

### Identification and paths

| Column | Meaning |
| --- | --- |
| case_id | Case-study identifier, for example 4POU. |
| model_id | Stable model identifier derived from the verified primary PDB filename. |
| prediction_method | AF2 or AF3. |
| af_version | AF2-Multimer version or alphafold3. |
| relaxation_status | unrelaxed, relaxed or as_predicted. |
| model_path | Repository-relative path to the primary PDB. |
| energy_path | Repository-relative path to the source .ene file. |
| confidence_path | Path to the AF2 log.txt or AF3 summary confidence JSON. |

### pyDock energies and rankings

| Column | Meaning |
| --- | --- |
| Ele | Electrostatic energy read from the .ene file. |
| Desolv | Desolvation energy read from the .ene file. |
| VDW | Full van der Waals term read from the .ene file. |
| pyDock-0.1VDW | Native `.ene` `Total` score, using 0.1 × VDW. |
| pyDock-1VDW | Recalculated energy using the full VDW contribution. |
| rank_pyDock-0.1VDW | Global ascending rank of pyDock-0.1VDW within the case. |
| rank_pyDock-1VDW | Global ascending rank of pyDock-1VDW within the case. |

### Confidence and combined ranking

| Column | Meaning |
| --- | --- |
| pLDDT | AF2 per-model pLDDT extracted from `log.txt`; normally absent from AF3 summary JSON. |
| pTM | Predicted TM score from the AF2 log or AF3 summary JSON. |
| ipTM | Interface predicted TM score from the AF2 log or AF3 summary JSON. |
| ranking_score | Native AF3 ranking score; normally absent for AF2 recycle records. |
| AF-MC | 0.8 × ipTM + 0.2 × pTM. |
| Z_AF-MC | Per-case Z-score of AF-MC. |
| Z_pyDock-0.1VDW | Per-case Z-score of pyDock-0.1VDW. |
| Z_pyDock-1VDW | Per-case Z-score of pyDock-1VDW. |
| Z_AF/pyDock-1VDW | Z_AF-MC minus Z_pyDock-1VDW. |
| rank_Z_AF/pyDock-1VDW | Descending rank of Z_AF/pyDock-1VDW. |

### AF2 and AF3 model metadata

| Column | Meaning |
| --- | --- |
| af2_filename_rank | AF2/ColabFold final-model position encoded in `_rank_NNN_` in the PDB filename; this is not a pyDock or combined rank. |
| model_number | AF2 model number. |
| seed | AF2 or local AF3 random seed parsed from the input record or filename; empty for AF3 Server. |
| recycle | AF2 recycling iteration. |
| sample | Local AF3 sample number. |
| tol | AF2 recycle tolerance when present in log.txt. |

Integer metadata and ranking fields are written without decimal suffixes.
Fields that do not apply to a prediction method remain empty; for example,
`af2_filename_rank`, `model_number` and `recycle` are empty for AF3 rather than being
assigned artificial values.

### Additional AF3 confidence fields

| Column | Meaning |
| --- | --- |
| fraction_disordered | AF3 predicted disordered fraction when present. |
| has_clash | AF3 clash indicator when present. |
| num_recycles | Optional AF3 recycle count; it can be absent from summary JSON. |
| chain_ids | Ordered unique AF3 chain identifiers, serialized as JSON in the CSV. |
| chain_ptm | AF3 per-chain pTM array, serialized as JSON. |
| chain_pair_iptm | AF3 chain-pair ipTM matrix, serialized as JSON. |
| chain_A_pTM | First chain pTM value, corresponding to chain A for the current 4POU layout. |
| chain_B_pTM | Second chain pTM value, corresponding to chain B for the current 4POU layout. |
| ipTM_A_vs_B | First off-diagonal chain-pair ipTM value for the current two-chain layout. |

### Diagnostics

| Column | Meaning |
| --- | --- |
| pdb_resolution | How the model PDB was found: ini or validated_suffix. |

## Limitations and practical notes

- AF3 Server CIF models must first be converted to PDB and scored with pyDock;
  section 3.3.2 performs the conversion before creating the pyDock jobs.
- No experimental structure, RMSD or CAPRI metrics are required.
- The notebook does not generate models, relax structures or execute pyDock.
- Method-specific empty values are expected. For example, pLDDT and tol are
  AF2-derived, while ranking_score and chain-level arrays are AF3-derived.
- The derived `chain_A_pTM`, `chain_B_pTM` and `ipTM_A_vs_B` fields assume the first
  two entries represent chains A and B. The original arrays are retained for
  cases with different or more complex chain organization.
- Re-running the notebook always overwrites the two CSV tables. Existing PNG
  files remain untouched when `GENERATE_PLOTS=False` or when there are no
  rankable models; otherwise the three plots are overwritten. Setting
  `DISPLAY_PLOTS=False` hides the inline figures without changing PNG output.
